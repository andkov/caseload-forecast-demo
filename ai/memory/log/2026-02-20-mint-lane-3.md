## Summary

Implemented `manipulation/3-mint-IS.R` — the single unified Mint lane that transforms Ellis parquet output into all model-ready artifacts consumed by the four Train tiers. Subsequently migrated all data artifacts from R-native `.rds` to Apache Parquet for cross-language compatibility with Python and Azure ML. Validated end-to-end via a dedicated inspection utility.

---

## Architecture Decision: Single Unified Mint

**Decision**: One Mint script producing one artifact set consumed by all 4 model tiers.

**Alternatives considered**: Separate `mint-a.R` and `mint-b.R` per model type.

**Rationale**:
- Prevents artifact drift (single `forge_hash` links all models to the same data slice)
- Aligns with cloud-native data asset patterns (Azure ML `TabularDataset`, Snowflake reference datasets)
- Simpler governance: one script to audit for train/test leakage
- Each Train tier selects what it needs; Mint doesn't decide what models exist

---

## Architecture Decision: Apache Parquet for Forge Artifacts

**Decision**: All data artifacts in `./data-private/derived/forge/` saved as `.parquet`. Fitted model objects (Train output) remain `.rds`.

**Rationale**:
- Columnar format reads efficiently in both R (`arrow::read_parquet`) and Python (`pd.read_parquet`)
- Azure ML `TabularDataset` natively consumes parquet without conversion step
- Self-describing schema (column names and types preserved)
- Future Tier 4 dynamic predictors may be sourced from Python pipelines — parquet is the natural exchange format

**ts objects not persisted**: `ts_train`, `ts_test`, `ts_full` are built in-memory during Mint for contract validation, but are NOT saved to disk. Train lane reconstructs them from `ds_*.parquet` with one line:
```r
ts_train <- ts(ds_train$y, start = c(year(min(ds_train$date)), month(min(ds_train$date))), frequency = 12)
```
This avoids R-specific serialization for the most commonly accessed data.

**Exception**: Fitted model objects (output of Train lane) remain `.rds` — R model objects (ARIMA, ETS, etc.) have no meaningful parquet representation.

## Static xreg Design Notes

- **Source**: `client_type_wide.parquet` (162 months: Apr 2012 – Sep 2025)
- **Variables**: `prop_etw_working`, `prop_etw_available`, `prop_etw_unavailable` (BFE excluded as complement to avoid collinearity)
- **Back-fill**: Pre–Apr 2012 months (84 months) carry earliest available proportions
- **Future**: Last observed proportions (Sep 2025) carried forward for 24-month forecast horizon
- **Proportions genuinely vary**: `prop_etw_working` 0.0817 → 0.0452; `prop_etw_available` 0.2098 → 0.4395 (substantial labor market shift over 2012–2025)
